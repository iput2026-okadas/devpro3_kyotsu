# Client / Server Unit Test シナリオ

Python の `unittest` を使用し、`Server/` の各処理を関数・クラス単位で確認します。
外部サービス、実ソケット、実際の時刻、サンプル CSV には依存せず、必要な箇所は
`unittest.mock` と一時ディレクトリを使用します。

## `Client/client.py`

### クライアントID引数

- `-i`（`--client-id`）、接続先ホスト、ポートを渡し、各値が解釈されることを確認する。
- クライアントID前後の空白が除去されることを確認する。
- クライアントIDの省略と空文字がエラーになることを確認する。

### クライアントIDを含むJSON送信

- センサー読み取り、ソケット、待機処理をモックする。
- 各送信JSON行に指定した `client_id` とセンサー値が含まれることを確認する。
- JSONの末尾に改行があり、指定したホストとポートへ接続することを確認する。

## `alert.py`

### 1. 警告レベルの Slack 通知

- `requests.post` をモックする。
- CO2 値 `1200`、通知レベル `warning` で `send_alert` を呼び出す。
- Webhook URL、タイムアウト `10`、注意文を含む JSON が渡されることを確認する。
- HTTP エラー確認の `raise_for_status` が呼ばれることを確認する。

### 2. 危険・回復レベルの Slack 通知

- `requests.post` をモックする。
- `danger` では警告文と CO2 値、`recover` では回復文と CO2 値が送信されることを
  確認する。
- どちらも Slack 用の `text` キーを持つことを確認する。

### 3. Webhook URL 未設定時の処理

- `SLACK_WEBHOOK_URL` を未設定にする。
- `send_alert` を呼び出す。
- 未設定を示すメッセージが標準出力へ表示され、HTTP 通信が行われないことを確認する。

### 4. 不明な通知レベルの処理

- 定義されていない通知レベルで `send_alert` を呼び出す。
- HTTP 通信が行われず、例外も発生しないことを確認する。

### 5. Slack 通知失敗時の処理

- `requests.post` または `raise_for_status` が `requests.RequestException` を送出するように
  モックする。
- `send_alert` を呼び出す。
- 例外が呼び出し元へ伝播せず、通知送信エラーが標準出力へ表示されることを確認する。

### 6. センサーデータの保持と部分更新

- `latest_data` を初期状態へ戻す。
- `temp`, `humid`, `light_percent` の一部だけを含むデータを順に渡す。
- 渡した項目だけが更新され、未指定の項目は以前の値を保持することを確認する。
- CO2 値がまだない場合は通知されないことを確認する。

### 7. CO2 状態の境界値と遷移

- CO2 値 `999`, `1000`, `1499`, `1500` をそれぞれ入力する。
- `999` は `normal`、`1000` と `1499` は `warning`、`1500` は `danger` と判定される
  ことを確認する。
- `normal` から `warning`、`warning` から `danger`、`danger` から `normal` へ変化した
  ときに、それぞれ1回だけ適切な通知が行われることを確認する。

### 8. 同一状態での重複通知防止

- 現在状態を `warning` または `danger` に設定する。
- 同じ状態に分類される CO2 値を複数回入力する。
- 状態が変化せず、Slack 通知も行われないことを確認する。

### 9. 不正な CO2 値の処理

- CO2 値として `None`、数値に変換できない文字列、空文字を入力する。
- 状態が変化せず、Slack 通知が行われないことを確認する。
- 数値に変換できない値では、エラーメッセージが標準出力へ表示されることを確認する。

## `server.py`

### 10. 受信データの行追加

- データ一覧を空にし、現在時刻を固定する。
- `client_id`, `temp`, `humid` を持つ辞書を `add_data` に渡す。
- ID、クライアントID、固定した時刻、気温、湿度の順で1行追加されることを確認する。
- 複数回追加した場合、ID が `0` から連番になることを確認する。

### 11. 必須項目がない受信データ

- `client_id`, `temp`, `humid` のいずれかがない辞書を `add_data` に渡す。
- `KeyError` が発生し、不完全な行がデータ一覧へ追加されないことを確認する。
- 空文字や文字列以外の `client_id` は `ValueError` になり、追加されないことを確認する。

### 12. CSV ファイルの保存

- `DATA_DIR` を一時ディレクトリへ差し替え、現在時刻を固定する。
- 複数行のデータを用意して `save_data` を呼び出す。
- `data-YYYYMMDDHHMMSS.csv` が作成されることを確認する。
- ヘッダーが `id,client_id,timestamp,temp,humid` で、全データが順序どおり UTF-8 で
  保存されることを確認する。

### 13. 空データの CSV 保存

- データ一覧を空にして `save_data` を呼び出す。
- ヘッダーだけを持つ CSV ファイルが作成されることを確認する。

### 14. TCP 受信データの処理

- ソケットの `recv` が有効な JSON バイト列を返した後、空バイト列を返すようにする。
- `add_data` と `alert.process_sensor_data` をモックして `recv_data1024` を呼び出す。
- デコードされた同一の辞書が両方の処理へ渡されることを確認する。
- 正常切断後にソケットが必ず閉じられることを確認する。

### 15. TCP 受信時の異常終了

- 不正な JSON、`ConnectionResetError`、予期しない受信例外をそれぞれ発生させる。
- 対応するエラーメッセージが標準出力へ表示されることを確認する。
- どの異常でもソケットが `finally` で閉じられることを確認する。

## `app.py`

### 16. 最新 CSV の初期選択

- `DATA_DIR` を複数のテスト用 CSV がある一時ディレクトリへ差し替える。
- Flask のテストクライアントで `/` へアクセスする。
- `data-*.csv` が新しい名前順で渡され、先頭の CSV が選択されることを確認する。
- CSV の列、行、行数がテンプレートへ正しく渡されることを確認する。
- `client_id` が各行の送信元として読み込まれることを確認する。

### 17. 指定した CSV の読み込み

- `file` クエリーパラメーターに一覧内の CSV 名を指定する。
- 指定したファイルの列と行が読み込まれ、選択中のファイル名として表示されることを
  確認する。

### 18. 気温・湿度の平均値

- 複数の `temp` と `humid` を持つ CSV を読み込む。
- それぞれの平均値が小数第1位へ丸められることを確認する。
- 空欄や数値に変換できない行が計算対象から除外されることを確認する。

### 19. 空または列不足の CSV

- ヘッダーだけの CSV、`temp` または `humid` 列がない CSV をそれぞれ読み込む。
- 例外が発生せず、対象データがない平均値は `0`、行数は `0` になることを確認する。

### 20. 存在しない CSV の指定

- 存在しないファイル名を `file` パラメーターへ指定する。
- ファイルが見つからない旨のエラーがテンプレートへ渡されることを確認する。
- 列と行が空のままであることを確認する。

### 21. データディレクトリ外のファイル指定

- `../`、絶対パスなど、`DATA_DIR` の外を指す値を `file` パラメーターへ指定する。
- ディレクトリ外のファイルが読み込まれず、エラーとして扱われることを確認する。

### 22. CSV が1件もない場合

- 空の一時ディレクトリを `DATA_DIR` に設定する。
- `/` へアクセスしても例外が発生しないことを確認する。
- ファイル一覧、選択ファイル、列、行が空で、平均値が `0` になることを確認する。

### 22.1. 従来形式CSVとの互換性

- `client_id` 列がない従来の4列CSVを読み込む。
- 列、行、平均値が例外なく読み込まれることを確認する。

## `analyze_environment.py`

### 23. 各センサー値の単独判定

- 気温、湿度、CO2、照度について、各しきい値の直前・一致・直後を入力する。
- 高温・低温、高湿度・低湿度、高 CO2、暗すぎる・明るすぎる状態が仕様どおり判定
  されることを確認する。
- 判定ごとに対応する原因と対策が返されることを確認する。

### 24. 複合環境の判定

- 高温かつ高湿度、高温かつ高 CO2、高湿度かつ高 CO2 の条件をそれぞれ満たす値を
  入力する。
- 単独判定に加えて、対応する複合原因と対策が返されることを確認する。
- 複数条件に同じ対策が含まれても、返却結果では重複しないことを確認する。

### 25. 快適な環境の判定

- どの警告条件にも該当しない値を入力する。
- 原因が「室内環境は快適」、対策が「現在の環境を維持する」だけになることを確認する。

## `environment_advisor.py`

### 26. 環境アドバイスの境界値

- 気温 `27` と `30`、湿度 `70`、CO2 `1000`、照度 `20` の境界値を個別に入力する。
- 各境界値で定義された原因と対策が返されることを確認する。
- どの条件にも該当しない場合は、快適な環境を維持する回答だけが返されることを
  確認する。

### 27. 複数の問題がある環境

- 高温、高湿度、高 CO2、低照度を同時に満たす値を入力する。
- 該当するすべての原因と対策が定義順で返されることを確認する。

## `chatbot.py`

### 28. チャットボットの初期化

- `analyze_environment` をモックして `EnvironmentChatBot` を生成する。
- センサー値がインスタンスへ保持されることを確認する。
- 4つのセンサー値が分析処理へ渡され、返された原因と対策が保持されることを確認する。

### 29. センサーごとの回答

- 気温、暑さ、湿度、CO2、明るさに関する質問をそれぞれ渡す。
- 対応する現在値と単位を含む回答が返されることを確認する。
- 英字の大文字・小文字が異なる CO2 の質問でも同じ回答になることを確認する。

### 30. 原因・解決策・状態の回答

- 原因、解決策、「どうすれば」、現在の状態に関する質問をそれぞれ渡す。
- 初期化時に分析した原因・対策が、見出しと箇条書きを伴って返されることを確認する。
- 状態の回答には4つのセンサー値と原因が含まれることを確認する。

### 31. 未対応の質問

- 既知のキーワードを含まない質問を渡す。
- 例外が発生せず、利用できる質問例が返されることを確認する。

## テストデータと実行方針

- テストは `tests/unit/test_*.py` に配置し、テストごとにグローバル状態を初期化する。
- Slack Webhook は必ずモックし、実際の URL や `Server/.env` を使用しない。
- CSV は `tempfile.TemporaryDirectory` などで作成し、`Server/data/` を変更しない。
- 時刻、ファイルパス、ソケット、標準出力は `unittest.mock` で差し替える。
- `server_test` の実ソケット待受や Flask のブラウザ表示は Unit Test の対象外とし、
  結合テストまたは `docs/test-e2e-scenario.md` の E2E テストで確認する。
- シナリオ 21 はディレクトリ外参照を防ぐためのセキュリティ要件であり、現行実装で失敗する
  場合は `app.py` の入力検証を追加してから成功させる。
